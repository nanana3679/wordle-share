# 문서 컨벤션

/docs 하위 문서를 작성·수정할 때 아래 규칙을 따른다.

## 파일명

- 기획/설계 문서: `SCREAMING_SNAKE_CASE.md` (예: `DATABASE_SCHEMA.md`)
- `feature/` 하위 피처 문서: `kebab-case.md` (예: `google-login-flow.md`)
- `issue/` 하위 이슈 문서: `kebab-case.md`

## 구조

- `# 제목` → `## 섹션` → 불릿 포인트 중심
- 서술형 문장은 최소화하고, 핵심만 짧게
- 한국어 기본, 기술 용어는 영어 그대로 사용

## 코드 예제

- 실제 구현 코드 삽입은 지양
- 구조/네이밍/시나리오를 보여주기 위한 간략한 예시는 허용
- 코드가 필요한 경우 `feature/` 하위에 별도 문서로 분리

## 길이

- 관행적으로 ~80줄 목표 (강제는 아님)
- 초과 시 주제별로 별도 파일로 분할 고려

## 문서 작성 시점

- 기능(Feature)을 완성할 때마다 `/docs` 또는 `/docs/feature`에 문서 작성
- 이슈 발생 시 `/docs/issue`에 문제 상황, 사고 흐름, 해결 과정을 기록

## 작업 전 확인

- 대화 시작 시 `/docs`에 어떤 문서가 있는지 먼저 살펴보고, 요구사항과 관련된 문서를 참고한다
- 기존 문서와 내용이 충돌하지 않는지 확인한 뒤 작성/수정한다
