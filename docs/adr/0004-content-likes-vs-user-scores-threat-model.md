# 0004. 콘텐츠 좋아요 vs 사용자 점수의 위협 모델 분리

## Status

Accepted

## Context

[0003](./0003-no-public-user-leaderboard.md)에서 사용자 점수 랭킹을 거절했다.
같은 논리로 "그럼 콘텐츠 좋아요 랭킹도 거절해야 하지 않나?" 질문이 자연스럽다.

두 시그널은 **위협 모델이 다르다**:

| | 점수 cheating | 좋아요 spam |
|---|---|---|
| 공격 단위 | 1인 자기 점수 부풀림 | 1인이 N개 콘텐츠 push |
| 신호 영향 | 본인 랭킹만 변동 = 신호 100% 무가치화 가능 | 진짜 사용자 합산에 비해 노이즈 작음 |
| 필요 액션 | 1회 시도/일 | N배 액션 (N개 덱마다) |
| robustness | 집단 통계 아님, 개인 행위 | 집단 시그널, 다수의 합 |

## Decision

- **사용자 점수 랭킹은 도입하지 않는다** — 신호 무가치화 위험
- **콘텐츠(덱) 좋아요 랭킹은 도입한다** — 메인 피드 Hot/좋아요순 탭으로 활용
- 두 시그널의 cheating-resistance 요건을 분리해서 평가
- 좋아요는 cheating-resistant 정도가 충분 — IP 단독 식별([0002](./0002-ip-hash-for-likes.md))로 처리 가능

## Consequences

- "랭킹 자체를 거부한다"는 단순 원칙은 폐기. 위협 모델별로 결정
- 메인 피드 Hot 탭 = log10(likes) + 시간 가중. Reddit식 정렬 도입 가능
- VPN 사이클 spam은 진짜 좋아요 분포 대비 무시 가능 수준 — 임계 도달 시에만 모더레이션 개입
- 향후 다른 시그널(공유 횟수·풀이 횟수 등) 도입 시도 같은 위협 모델 분석부터 수행
- 디자인 원칙 #8b로 박제: "콘텐츠 랭킹은 별개 위협 모델"
