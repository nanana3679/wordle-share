# 기여 가이드

## PR 규모

**원칙: diff 400줄 이하를 목표**. lock 파일·자동 생성물 제외.

| 규모 | 판단 |
|---|---|
| ~200줄 | 이상적 — 리뷰어가 한 번에 집중해 끝냄 |
| ~400줄 | 허용 — 신기능/리팩토링 |
| 400~800줄 | 경계선 — 자연 분할 가능성 재검토 |
| 800줄+ | 원칙상 분할. 불가피하면 PR 본문에 이유 명시 |

예외 허용: 100% 추가 변경(기존 로직에 영향 없음) + 단일 논리 단위 + 쪼개면 중간 상태가 무의미한 경우.

## 자연 분할 패턴

큰 기능은 **스키마/인프라 → 로직 → UI** 순으로 쪼개는 게 기본:

1. **DB/스키마 준비** — 마이그레이션, 타입, RLS 정책
2. **서버 로직** — server actions, lib/, 검증 함수
3. **UI/스크립트** — 컴포넌트, 스크립트, 페이지

각 PR이 독립적으로 머지 가능해야 하며, **이전 PR이 머지된 상태에서 테스트 가능**해야 함.

## 문서·코드 분리 가능

다음 경우 문서만 별도 PR로 올려도 됨:

- **인덱스 재구성, 용어 통일** 같은 범용 문서 작업 (`docs/todo.md` 재정리 등)
- **기존 기능의 설계 회고** (`docs/issue/*.md`)
- **여러 기능에 영향 주는 가이드** (이 `CONTRIBUTING.md` 같은 것)

단 **신기능 feature 문서**(`docs/feature/<name>.md`)는 해당 기능 코드와 같은 PR에 포함. 코드 없이 문서만 있으면 "스펙 초안"이 되어 구현과 어긋날 위험.

## 커밋 단위

- 한 커밋 = 한 논리적 변경
- 프롬프트 튜닝·버그 수정이 여러 번 반복되면 `git rebase -i`로 squash 후 올림
- 커밋 메시지는 "왜"를 설명 (무엇은 diff로 보면 됨)

## 리뷰 응답

- CodeRabbit 등 봇 지적은 **선별적으로** 반영. 의도된 동작이면 댓글로 응답
- "Minor" 중에서도 trade-off가 있는 항목은 판단 근거 명시

## PR 본문 템플릿

```markdown
## 요약
- 1~3 bullet

## 변경
- 파일별 주요 변경

## 테스트 계획
- [ ] 체크리스트

## 후속/블로커
- 다음 PR에서 할 일
```
